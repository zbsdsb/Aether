# Aether 模型操练场设计

日期: 2026-04-09
状态: 已确认设计，待写实现计划

## 1. 背景

当前 Aether 已具备模型目录、Provider 级模型查看、上游模型抓取、以及带故障转移信息的模型测试能力，但缺少一个面向“真实调参与协议验证”的统一页面。

用户当前需要的是一个类似 metapi / New API Playground 的“模型操练场”：

- 可以像聊天页一样做多轮测试
- 可以切换协议与常见生成参数
- 可以直接看到请求预览、真实请求和真实响应
- 不仅能测“平台里的全局模型”，也能单独测某个渠道商的模型
- 在渠道模式下，即使某个协议当前还没正式配置，也希望能先手动探测它是否可用，从而指导后续补配置

## 2. 目标

- 新增一个独立的“模型操练场”页面，而不是继续把测试能力塞进现有模型页或 Provider 页
- 同一页面同时支持：
  - `全局模型` 测试
  - `渠道模型` 测试
- 支持以下协议模式切换：
  - `OpenAI Chat`
  - `OpenAI Responses`
  - `Claude`
  - `Gemini Native`
- 页面采用三栏结构：
  - 左栏参数区
  - 中栏对话区
  - 右栏调试区（`预览 / 请求 / 响应`）
- 让右栏明确区分：
  - 正式配置测试
  - 临时协议探测
- 第一版尽量复用 Aether 当前已有测试链路、模型数据源与协议元数据能力

## 3. 非目标

- 不做外部临时 `base_url + api_key` 直连测试
- 不做文件上传、多模态附件、工具调用编排等重型 Playground 能力
- 不在第一版中重做现有 Provider 配置流程
- 不把当前页面扩成新的全功能聊天产品页
- 不自动改写 Provider 配置；即使探测成功，也只提供“去补配置”的引导，不直接落库

## 4. 已确认决策

### 4.1 页面形态

采用单页统一操练场，而不是按协议拆成多张页。

原因：

- 用户的核心需求是“像 metapi 一样在一页里切来源、切协议、调参数、看请求”
- Aether 当前已有协议能力与测试链路，适合做统一编排
- 后续扩充参数和调试能力时，单页状态模型更稳定

### 4.2 模型来源分为两种

左栏顶部先切换测试来源：

- `全局模型`
- `渠道模型`

语义分别为：

- `全局模型`：按 Aether 的平台真实路由行为测试目标模型
- `渠道模型`：绕过全局路由，直接针对某个 Provider / 模型做测试

### 4.3 渠道模型允许手动切协议

在 `渠道模型` 模式下，协议不是只读，而是允许用户手动切换。

原因：

- 用户希望借此快速验证“当前未配置但实际上可能可用”的协议
- 如果探测成功，可直接反向指导 Provider 补协议配置

但必须明确区分 3 种状态：

- `已配置`
- `未配置，可试测`
- `未验证，高风险`

### 4.4 右侧调试区必须保留

第一版保留右侧 `预览 / 请求 / 响应` 三个标签。

原因：

- 这是操练场区别于普通聊天页的关键价值
- 用户需要真实看到协议映射结果、请求体和响应体，才能判断“协议没配”还是“协议不通”

### 4.5 第一版允许分阶段落地

实现可以分为：

- `P0`：页面骨架、来源切换、协议切换、参数区、中栏消息区、右栏调试区，以及复用现有测试链路跑通 `全局模型` 和 `已配置协议的渠道模型`
- `P1`：补齐 `渠道模型 + 未配置协议` 的真实“临时协议探测”能力

这样即使第一轮范围需要控制，也不会把整页价值完全砍掉。

## 5. 用户体验设计

### 5.1 路由与入口

页面作为独立视图存在：

- 用户侧入口：`/dashboard/playground`
- 管理侧入口：`/admin/playground`

两者指向同一个前端页面组件，避免维护两套实现。

导航名称统一为：

- `模型操练场`

### 5.2 顶部状态卡

页面顶部保留 4 张状态卡：

- `可测模型数`
- `当前目标`
- `当前协议`
- `当前会话轮次`

这些卡片不承担筛选能力，只负责快速给出当前测试上下文。

### 5.3 左栏：设置区

左栏分为 5 个层次。

#### A. 来源切换

- `全局模型`
- `渠道模型`

#### B. 目标选择

当选择 `全局模型` 时：

- 全局模型搜索框
- 全局模型下拉

当选择 `渠道模型` 时：

- 渠道商搜索框
- 渠道商下拉
- 模型搜索框
- 模型下拉

#### C. 协议切换

显示 4 个协议模式：

- `OpenAI Chat`
- `OpenAI Responses`
- `Claude`
- `Gemini Native`

在 `渠道模型` 模式下，每个协议项旁显示当前状态徽标：

- `已配置`
- `未配置，可试测`
- `高风险`

#### D. 参数区

第一版参数：

- `System Prompt`
- `流式输出`
- `思考深度`
- `temperature`
- `max output tokens`
- `top_p`

参数项不要求所有协议完全一致：

- 不支持或可能被忽略的参数允许继续填写
- 但要显示轻量 warning，而不是静默吞掉

例如：

- Claude 模式下 `temperature` 可能被忽略
- Gemini Native 下 `system prompt` 会转换到 `systemInstruction`

#### E. 高级请求区

默认折叠，包含：

- `附加请求头`
- `附加请求体覆盖`

这里的目标不是开放完整自由拼装，而是给操练场提供一个轻量的“补字段 / 试字段”入口。

#### F. 主操作

底部保留按钮：

- `发送`
- `停止`
- `清空`
- `重试上次`

若当前为 `渠道模型 + 未配置协议`，则按钮旁额外提示：

- `临时探测`

### 5.4 中栏：对话区

中栏负责承载实际测试过程。

顶部状态显示：

- `就绪`
- `请求中`
- `流式传输中`
- `失败`

消息区支持多轮消息：

- `system`
- `user`
- `assistant`
- `thinking / reasoning`（若可用则折叠展示）

用户消息支持轻操作：

- `重试`
- `复制`
- `编辑`
- `删除`

底部是多行输入框与发送区。

错误时，中栏插入一条失败结果卡片，最少显示：

- 协议
- 目标模型
- Provider（若有）
- 错误摘要

### 5.5 右栏：调试区

右栏保留 3 个标签：

- `预览`
- `请求`
- `响应`

#### 预览

展示协议标准化后的请求结构。

例如：

- OpenAI Chat -> `messages`
- OpenAI Responses -> `input`
- Claude -> `messages + system`
- Gemini Native -> `contents + systemInstruction`

#### 请求

展示真正发送出去的 body / headers，并补上下文：

- 当前来源模式
- 当前协议
- `正式配置测试` 还是 `临时协议探测`
- endpoint / base_url
- request id

#### 响应

展示：

- 原始响应 JSON
- 状态码
- 错误体
- 若为流式，则显示片段摘要与最终聚合结果

### 5.6 探测成功后的引导

当 `渠道模型 + 未配置协议` 探测成功时，页面提供一个次级动作：

- `去补协议配置`

它不直接改配置，只负责把当前成功探测的关键信息交给后续配置流程或复制建议文本。

## 6. 前端状态模型

建议将页面状态拆成 4 层，而不是做成一个大表单对象。

### 6.1 来源层

- `sourceMode = global | provider`

### 6.2 目标层

全局模式：

- `globalModelId`
- `globalModelName`

渠道模式：

- `providerId`
- `providerName`
- `targetModelName`
- `protocolProbeMode = configured | probe`

### 6.3 协议层

- `protocol = openai-chat | openai-responses | claude | gemini-native`

该层直接驱动：

- 参数区显示逻辑
- 请求预览生成
- 消息格式转换
- 实际执行分支

### 6.4 会话层

- `messages`
- `systemPrompt`
- `requestOverrides`
- `lastRequestPreview`
- `lastRawRequest`
- `lastRawResponse`
- `streamState`
- `runStatus`

这样页面可以稳定支持多轮测试、重试、协议切换后的消息保留与右栏回看。

## 7. 数据源与执行策略

### 7.1 模型数据源

前端复用现有数据源：

- 全局模型列表：现有 GlobalModel API
- 渠道模型列表：现有 Provider 级模型 API / 上游模型查询能力

这意味着页面不需要额外先造一套模型目录后端。

### 7.2 P0：复用现有测试链路

第一阶段优先复用现有 `testModelFailover` 思路：

- `全局模型` -> `mode=global`
- `渠道模型 + 已配置协议` -> `mode=direct`

这两条路径都属于“系统当前真实可调度能力”的测试。

### 7.3 P1：未配置协议的临时探测

`渠道模型 + 未配置协议` 不应伪装成已配置直连。

建议新增一条更适合 Playground 的探测接口，允许：

- 指定 `provider_id`
- 指定 `model_name`
- 指定 `api_format`
- 使用该 Provider 现有 base URL / key / connector 生成一次临时测试请求
- 不落库、不改配置，只返回探测结果与真实请求信息

这条链路的目标是“判断协议可能可用”，而不是“声明系统已正式接入”。

### 7.4 正式测试与探测测试的区分

页面上必须显式区分：

- `正式配置测试`
- `临时协议探测`

并且在调试区固定显示提示：

- `本次为临时协议探测，请求成功不代表该协议已正式配置到该渠道`

## 8. 组件拆分建议

不建议把所有逻辑堆到单个视图文件中。

建议最少拆成：

- `PlaygroundView`
  - 页面总编排
- `PlaygroundSetupPanel`
  - 左栏设置
- `PlaygroundConversationPanel`
  - 中栏消息与输入
- `PlaygroundDebugPanel`
  - 右栏调试面板
- `usePlaygroundState`
  - 页面状态编排
- `usePlaygroundPreview`
  - 请求预览生成
- `usePlaygroundRunner`
  - 实际测试执行

这样后续补 `P1` 探测能力时，不会把整页状态机继续搅在一起。

## 9. 实现分期

### P0

- 路由与导航入口
- 三栏页面骨架
- 两种来源切换
- 四种协议切换
- 左栏参数区
- 中栏对话区
- 右栏 `预览 / 请求 / 响应`
- 复用现有测试链路跑通：
  - `全局模型`
  - `渠道模型 + 已配置协议`

### P1

- `渠道模型 + 未配置协议` 的真实临时探测接口
- 右栏对探测模式的更完整展示
- 探测成功后的“去补协议配置”引导

## 10. 测试设计

### 10.1 前端

至少覆盖：

- 来源切换后选择器显示正确
- 协议切换后参数区和请求预览联动正确
- `已配置 / 未配置，可试测 / 高风险` 徽标判定
- 请求预览生成正确
- 中栏消息增删改重试逻辑
- 右栏 `预览 / 请求 / 响应` 切换

### 10.2 类型与构建

至少执行：

- `npm exec --yes vue-tsc -- --noEmit`

### 10.3 手工验证

至少覆盖：

- 全局模型模式发送一次成功请求
- 渠道模型模式发送一次成功请求
- 已配置协议成功 / 失败各一例
- 未配置协议状态展示正确
- 右栏请求与响应内容正确刷新

若 `P1` 一起实现，则额外验证：

- 临时协议探测成功一例
- 临时协议探测失败一例
- 成功后引导文案与状态标记正确

## 11. 风险与边界

- 最容易膨胀的点是“未配置协议探测”，它会把需求从纯前端页拉成前后端联动
- 协议切换后并非所有参数都完全等价，必须允许 warning 而不是伪装成全统一模型
- 若把探测成功错误地表述成“已接入”，会直接误导用户后续配置判断
- 三栏操练场的状态机会比普通表单页复杂，必须通过 composable 和子组件拆分控制复杂度

## 12. 推荐结论

采用“单页统一操练场 + 双来源模型选择 + 协议感知调试面板 + P0/P1 分期”的方案。

这是当前最平衡的路线：

- 足够接近 metapi / New API 的操练体验
- 可以复用 Aether 当前已有模型、协议和测试能力
- 能先落一版有真实价值的 Playground
- 同时为“未配置协议探测”留出明确扩展位，而不把第一版范围无限放大
