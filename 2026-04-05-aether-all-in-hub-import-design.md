# Aether 支持 All-in-Hub 导入的方案构思

日期: 2026-04-05
状态: 草案，待调研
适用范围: `Aether` 渠道商导入、站点补钥、迁移任务设计

## 1. 背景

当前希望让 `Aether` 支持从 `all-in-hub` 导出文件中自动导入渠道商配置，减少人工在后台逐个创建 `Provider / Endpoint / Key` 的成本。

初步调研后，已经明确以下事实：

- `Aether` 本身已经有完整的渠道商管理模型，核心结构是：
  - `Provider`
  - `ProviderEndpoint`
  - `ProviderAPIKey`
- `metapi` 已经具备 `all-in-hub` 备份导入能力，但导入数据并不总是包含可直接使用的明文 API Key。
- 某些上游站点的 token 列表接口只返回脱敏值，导致即使能登录、能列出 token，也无法直接恢复旧 token 明文。

因此，这个需求不应定义为“把导出文件直接塞进 Aether”，而应定义为：

1. 用导出文件完成结构化建模和基础导入。
2. 对缺失明文 Key 的记录，再走自动补全或替换式迁移。

## 2. 结论摘要

推荐主线方案：

- 直接给 `Aether` 增加正式的导入能力。
- 浏览器自动化不作为主线导入实现，而作为“补钥 worker”或“迁移辅助器”。

不推荐主线方案：

- 仅靠 `opencli` 或浏览器自动填写后台表单完成全部导入。

原因：

- 导入对象本质是结构化配置，不是纯 UI 录入问题。
- 仅靠浏览器自动填表很难做幂等、预览、失败定位、任务状态和批量重试。
- `Aether` 已经具备结构化 CRUD 基座，更适合承接正式导入能力。

## 3. 核心判断

### 3.1 为什么不能只依赖导入文件

`all-in-hub` 导出里的数据来源并不统一：

- 有些记录会带 `baseUrl + apiKey`
- 有些记录只有 `site_url + access_token/sessionCookie`
- 有些站点可以通过登录态继续查询 token 列表
- 但不少站点只返回脱敏 token，无法直接拿到明文

因此，“导入文件 -> 直接建 Aether Key”只能覆盖一部分情况。

### 3.2 为什么不能把“自动拉 token 列表”当成完整解法

即使能通过登录态访问上游 token 管理接口，也未必能拿到可用明文：

- 上游可能只返回脱敏值
- 页面列表可能也只展示脱敏值
- 旧 token 的完整明文可能只在创建当次展示一次

所以真正有效的迁移策略，不是“恢复旧 Key”，而是：

- 能直接导入明文的，直接导入
- 不能恢复明文的，优先生成 replacement Key
- replacement Key 生成成功并验通后，再移除旧 Key

## 4. 推荐目标模型

导入后，`Aether` 内部建议维持以下映射：

- 同一站点同一接口根地址 -> 一个 `Provider`
- 该站点支持的接口协议 -> 一个或多个 `ProviderEndpoint`
- 该站点下的多个令牌 -> 多个 `ProviderAPIKey`

站点识别建议优先基于：

- `baseUrl`
- `site_url`

统一归一化为同一个站点 origin，再结合平台类型做聚合。

简化理解：

- 同 origin: 一个 Provider，多 Key
- 不同 origin: 多个 Provider

## 5. 主方案: 结构化导入 + 补钥能力

### 5.1 第一阶段: 静态导入

目标：

- 先把导出文件中能够确定的结构化配置导入 `Aether`

建议动作：

- 根据 `baseUrl/site_url` 聚合站点
- 创建 `Provider`
- 根据站点协议创建 `Endpoint`
- 对导出文件中已有明文 `apiKey` 的记录，直接创建 `ProviderAPIKey`
- 对只有 `access_token/sessionCookie` 的记录，不直接伪装成可用 Key，而是记录为“待补钥来源”

### 5.2 第二阶段: 自动补钥

目标：

- 对缺少明文 Key 的记录，尝试自动补全

补全顺序建议如下：

1. 尝试读取站点现有 token 列表
2. 如果能拿到明文，直接导入
3. 如果只能拿到脱敏值，尝试登录站点并创建 replacement Key
4. 在 replacement Key 创建当次抓取明文
5. 成功后回填 `Aether`

### 5.3 第三阶段: 人工兜底

只有在以下场景下才进入人工补录：

- 站点 token 列表只返回脱敏值
- 创建 token 的 API 不返回明文
- 页面创建流程也抓不到明文
- 站点权限不足或登录态失效

## 6. 替换式迁移策略

这是当前最推荐的迁移模式。

### 6.1 不推荐的危险流程

不建议：

1. 先删除旧 Key
2. 再创建新 Key

风险：

- 新 Key 创建失败会直接中断业务
- 明文抓取失败会导致无法回填
- 分组、状态、默认路由可能不同步

### 6.2 推荐流程

推荐：

1. 读取旧 Key 元数据
2. 创建 replacement Key
3. 创建当次抓取明文
4. 回填到 `Aether`
5. 做最小可用性验证
6. 验证通过后删除旧 Key

### 6.3 需要保留的旧 Key 元数据

即使旧 Key 明文不可恢复，以下信息通常仍应尽量保持：

- 名称
- 分组
- 启用状态
- 默认/非默认角色
- 必要的模型限制或标签

### 6.4 命名策略

默认策略：

- 优先沿用旧 Key 的原名称

冲突策略：

- 若站点不允许重名，可临时使用后缀：
  - `xxx-migrating`
  - `xxx-reissued`

切换完成后：

- 如有必要，再回写为原名称

## 7. 浏览器自动化的角色边界

浏览器自动化建议只承担“补钥”职责，不承担完整导入职责。

适合浏览器自动化的动作：

- 登录站点
- 进入 token 管理页
- 读取旧 token 的元数据
- 新建 replacement Key
- 在创建完成弹窗或结果页抓取明文
- 删除旧 Key

不建议浏览器自动化承担的动作：

- 在 `Aether` 后台逐字段填完整个 Provider/Endpoint/Key 表单
- 作为唯一导入链路

原因：

- 站点页面自动化比后台结构化导入更脆弱
- 维护成本高
- 不利于任务状态管理和失败重试

## 8. 推荐的模块拆分

建议拆成三块：

### 8.1 Aether 导入模块

职责：

- 解析 `all-in-hub` 导出文件
- 聚合站点
- 创建 `Provider / Endpoint`
- 导入已有明文 Key
- 生成待补钥任务

### 8.2 补钥 Worker

职责：

- 使用站点登录态进入上游
- 判断是否能读取旧 Key 明文
- 明文不可取时创建 replacement Key
- 抓取创建当次明文
- 回填导入结果

### 8.3 迁移报告模块

职责：

- 输出每个站点/账号/Key 的迁移状态
- 标明以下状态：
  - 已直接导入
  - 已自动补钥
  - 已替换成功
  - 待人工处理
  - 失败

## 9. 推荐的数据状态

为了避免把“不完整来源记录”和“真正可用的 Aether Key”混在一起，建议有独立的迁移状态概念。

最小状态建议：

- `imported_ready`
- `pending_discovery`
- `pending_reissue`
- `pending_manual`
- `verified`
- `failed`

如果后续做成正式任务系统，可再补：

- 任务 ID
- 错误原因
- 最近一次尝试时间
- 是否允许重试

## 10. 关键风险

### 10.1 旧 Key 明文不可恢复

这是当前最大现实约束。

应对：

- 不以“恢复旧 Key 明文”为主目标
- 以“生成等价 replacement Key”作为主路径

### 10.2 上游站点创建接口不返回明文

应对：

- 检查创建响应是否含一次性明文
- 若 API 没有，尝试页面自动化抓取
- 若仍没有，则只能人工补录

### 10.3 页面自动化脆弱

应对：

- 页面自动化只承担补钥环节
- 对站点适配器做分层，不要写成一套巨大通用脚本

### 10.4 替换过程影响业务

应对：

- 先建新并验通，再删旧
- 不走“先删旧再建新”

## 11. 建议的调研清单

下一步调研应优先回答以下问题：

1. 常见上游站点的 token 创建接口，是否在响应中返回一次性明文
2. 如果接口不返回，页面创建成功后是否展示一次性明文
3. token 列表页返回的 `name/group/status` 是否稳定可读
4. 是否允许同名 token
5. 是否支持通过 API 删除旧 token
6. 是否存在验证码、人机校验或二次确认，影响浏览器自动化

## 12. 当前建议结论

当前阶段最合理的方向是：

- 在 `Aether` 中增加正式的 `all-in-hub` 导入能力
- 同时设计“站点补钥 / replacement Key 迁移”能力
- 把浏览器自动化降级为补钥辅助，而不是主导入链路

一句话总结：

不是“恢复旧 Key”，而是“结构化导入 + 安全替换旧 Key”。

## 13. 暂未定稿的问题

以下问题建议在调研后再定稿：

- replacement Key 的命名冲突策略是否统一
- 哪些站点走 API 创建，哪些站点必须走页面自动化
- 登录态是否允许短期缓存，还是只在任务执行期间内存使用
- 导入任务是否要先做 dry-run 预览
